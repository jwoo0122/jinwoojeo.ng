블로그를 (또) 갈아엎었다. 레이아웃은 크게 달라지지 않았지만, 글꼴과 색상을 바꿨다. 더 큰 변화는 코드에 있는데, Astro와 Tailwind를 완전히 떼어내고 순정의 html / css / javascript만 남겼다. 가장 처음 만들었던 블로그가 gatsby의 GraphQL 기반이었던 걸 생각하면 굉장히 많은 boilerplate를 덜어낸 셈이다. Astro를 쓸 때마다 생각했다. 고작 Markdown 문서 몇 개 보여주는 페이지를 만들기 위해 프레임워크가 필요한가? 대단한 스타일링을 할 것도 아닌데 Tailwind 문서를 붙잡아야 할까?

지금의 블로그는 Markdown 파일로부터 간단한 변환 스크립트를 통해 html을 만드는 구조로 되어 있다. 사실 흐름으로 보면 Astro를 사용할 때랑 대단한 차이는 없다. Markdown을 선택한 이상 어쩔 수 없다, 브라우저가 Markdown을 네이티브로 이해하지 못하니까. 다음엔 이조차 없애고 html을 진짜 "Markup" 서식으로 활용하는 습관을 길러볼까 한다.

# 말을 걸어오는 것들

나는 의도가 노골적으로 드러나는 물건들을 좋아한다. 영화나 만화 같은 작품을 감상할 때도 창작자가 의도하는 경험이 분명하게 전달되는 것이 좋다. 최근에는 \<유전\>을 봤는데, 공포물에 내성이 없는 나에게 쉽지 않은 작품이었지만 잘 만든 작품이라는 건 분명하게 느낄 수 있었다. 영화 전반이 일관된 (음산한) 분위기를 바탕에 깔고, 기괴한 오컬트 요소들에 관객을 지속적으로 노출시켜서 마치 실제로 그런 일이 일어났을 것 같은 기분을 들게 한다. 그렇게 관객을 끌어들이는 방식에서 극단적이고 강렬한 이미지보다 훨씬 만들기 어려운 "분위기"를 사용했다는 점이 더욱 잘 만든 작품이라는 생각을 들게 했다.

내가 Apple의 제품을 다른 전자기기에 비해 선호하는 것도 그런 연유다. 단순히 산업 디자인적으로 미감과 QC가 뛰어난 것 이전의 문제이다. 제품 간에 편차가 있겠지만, Apple은 자사의 제품이 만들어진 이유나 사용 의도를 아주 강렬하고 정확하게 전달한다. 많은 사람들이 Vision Pro에 회의감을 느끼고 나 또한 Vision Pro가 성공한 제품이라고 생각하지는 않는다. 특히 직접 착용해보기 전엔 거진 망한 제품이라고 생각했다. "공간 컴퓨팅"이라는 캐치 프레이즈도 그저 말장난인 줄 알았다. 하지만 직접 써보고 나서는 생각이 바뀌었다. 당장의 물건에는 하드웨어 기술의 한계가 있다. 그럼에도 불구하고, Apple이 가진 목표는 이것을 거의 하루종일 착용한 채로 현실과 가상의 경계를 허무는 경험이라는 걸 직관적으로 알 수 있었다. 그래서 단순한 엔터테인먼트나 사무용 제품이 아닌 "공간 컴퓨팅"이라는 말이 더 어울리는 것 같았다.

비슷하게 내가 매력을 느끼는 철학은 Unix Philosophy이다. 하나의 일을 확실하게 해결하는 소프트웨어를 만들고, 그런 소프트웨어를 조합하여 더 복잡한 문제를 해결하는 방법론이 아주 우아하다고 생각한다. awk나 sed, vim, grep 등 다양한 소프트웨어를 접하고 사용법을 익히다보면 문제를 적절하게 분해하는 시각을 익히게 되기도 한다. 한동안 그런 우아함을 직접 체험하고 익숙해지고자 터미널 중심 워크플로우를 설계해본 적도 있다.

# 글을 쓰기로 했다면

같은 원리로, 내 블로그의 목적은 의사 전달이다. 생각한 것을 설득력 있고 올바르게 정리하는 것이 글쓴이로서의 내 역할이라면, 작성된 글을 읽을 때 주의가 분산되지 않고 문해 자체에 집중할 수 있도록 만드는 것이 디자이너 혹은 엔지니어로서 나의 역할이다. 그래서 폰트와 레이아웃을 교체했다. 이전 레이아웃은 웹 초창기 홈페이지의 레이아웃을 오마주하려는 시도였는데, 애매하게 모던 타이포그래피와 섞이면서 의도 전달이 제대로 되지 않았던 것 같다. 그래서 이번엔 무명 소설가의 아지트를 컨셉으로 디자인을 엎어보았다.

기술적으로도 군더더기를 없앴다. Astro는 정적 웹사이트 빌딩에 최적화된 좋은 프레임워크이지만 지금의 내 블로그에는 overkill이라고 생각한다. 그도 그럴 것이 island architecture의 이점을 누릴 것도 없고, ui library agnostic한 컨셉이 필요하지도 않기 때문이다. 글쓰기에 필요한 핵심은 Markdown 문법을 잘 지원하며 확장에 용이할 것, 서식 수정이 간단할 것이었다. 그래서 모든 프레임워크 지원을 전부 걷어내고, Markdown to HTML converter만 적당히 정리했다.

글을 쓰는 과정 자체도 마찬가지다. 나는 다소 현학적인 표현을 채택하는 버릇이 있다. 글의 질이란 논리의 전개에서 오는 것인데, 필요 이상으로 압축적인 용어나 사어 등을 끌어와 있어보이는 글처럼 꾸미려는 것 같다. 그래서 앞으로는 좀 더 적절하고 이해하기 쉬운 문체를 추구하려 한다.

---

마지막으로, 이 converting 구조를 적당히 정리해서 블로그 생성 엔진으로 만들어볼까 싶다. 서두에 html native 블로그 운영을 하고 싶다고 했지만, markdown의 편리함을 이기기는 힘들 것 같다. 여하튼 누군가 잘 써준다면 재미있겠다.
